## Eval C — Known Failure Modes & Fixes

| ID | Failure | Root Cause | Fix Applied | Verify |
|----|---------|-----------|-------------|--------|
| C01 | Song skips after 10s with no player | `playback_controller` waits 10s for `start_song()` call; headless has no splash player | home.html HLS fetch of `.m3u8` triggers `start_song()` — user must open page within 10s | Song stays alive once browser connects |
| C02 | WebSocket 500 on upgrade | `gevent.pywsgi.WSGIServer` without `WebSocketHandler` | Added `handler_class=WebSocketHandler` to `app.py` line 241 | No `RuntimeError: gevent-websocket` in log |
| C03 | Video black in headless Chromium | Chromium headless shell (open-source) excludes proprietary H.264 codec | Use Firefox Playwright for visual evals | `video.readyState == 4` and `videoWidth > 0` in Firefox |
| C04 | HLS `bufferAddCodecError` in headless Chrome | Same as C03 | Same as C03 — eval tooling limitation, not user-facing | Pass in Firefox |
| C05 | Transpose eval timing | `transpose_current()` re-enqueues + skips; song restarts in ~10s | Wait ≥15s after `/transpose/{n}` before polling | `now_playing_transpose == n` after 15s |
| C06 | Pause eval timing | Socket race; `is_paused` state flips but poll catches old value | Wait ≥3s after `/pause` before polling | `is_paused` correctly reflects state |
| C07 | `POST /download` wrong schema | Old code sent `{"url":...,"user":...}` — schema requires `song_url`, `song_added_by`, `song_title`, `queue` | Updated eval script to use correct schema | HTTP 200 + `{"status":"ok"}` |
| C08 | Subtitles null for YT songs | YouTube `.mp4` downloads have no sidecar `.ass` file | Expected behaviour; SubtitlesOctopus wired and activates for CDG/ASS songs | `subtitle_url` non-null on CDG/ASS songs |
| C09 | HLS `startPosition: 0` stalls | With `EVENT` playlist type, `startPosition: 0` can cause HLS.js to not buffer | Changed to `startPosition: -1` (let HLS.js decide) | Video buffers within 5s |
| C10 | Video autoplay blocked (browser policy) | Browsers require user gesture before `video.play()` | `video.play().catch(warn)` — video starts on first user interaction | Click any button → video begins |
| C11 | `pikaraoke: command not found` | Script installed to venv not on PATH | Use `python -m pikaraoke.app` instead | App starts |
| C12 | SocketIO `JSON.parse` double parse | `$.get` with `application/json` response — jQuery doesn't auto-parse with `$.get`, only `$.getJSON` | `JSON.parse(data)` is correct with `$.get` | Song title shows correctly on page load |

## Hard Wins — Do Not Regress

These were hard to get right. Protect them.

1. **`handler_class=WebSocketHandler`** in `app.py` WSGIServer — without this SocketIO WebSocket fails silently with 500, falls back to long-polling which is slower and noisier.

2. **`attachMedia` before `loadSource`** in HLS.js setup — opposite order causes `MANIFEST_PARSED` to not fire in some browsers.

3. **`startPosition: -1`** not `0` — `0` causes HLS.js to stall on EVENT playlists in some configurations.

4. **`now_playing_subtitle_url`** is always null for YouTube downloads — this is correct, not a bug. Only CDG/ASS songs produce subtitle URLs via `/subtitle/{id}`.

5. **`start_song()` called by m3u8 fetch** — when `home.html` browser fetches the HLS playlist, `stream_playlist()` calls `k.playback_controller.start_song()`. This is the mechanism that prevents the 10s timeout in headless mode. If the browser doesn't connect within 10s, the song is skipped.

6. **Download API schema** — `POST /download` requires: `song_url`, `song_added_by`, `song_title`, `queue` (bool). NOT `url` or `user`.

7. **VP8/VP9/AV1 all available in ffmpeg** — if H.264 playback is ever needed in headless Chromium for testing, transcode with `-vcodec libvpx-vp9`. Full Chrome/Firefox on user devices already play H.264 natively.
